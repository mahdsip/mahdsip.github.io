REST

Representational State Transfer

Es un tipo de arquitectura basada en recursos, aunque si va acompañada de la definición de un contrato de servicio puede ser considerada un tipo de arquitectura SOA de tipo recurso. Se usa para describir cualquier interfaz web simple que utiliza tipicamente XML y HTTP. Se basa en la existencia de recursos accesibles atraves de URIs. Para manipular los recursos se comunican via HTTP e intercambian reprsentaciones de estos recursos. En RPC tu tienes las operaciones que puedes llevar a cabo , en REST tu obtienes el recurso (en XML u otro leguaje) y se operaria sobre el para luego subirlo con un HTTP put. REST implica el cambio de estado de un recurso pero sin definir la operación. Las peticiones que se pueden realizar a un recurso son las limitadas por el protocolo HTTP: POST para guardar datos, GET para recoger el recurso PUT para modificarlo y DELETE para borrarlo. Cuando el problema en cuestion que tratamos solucionar esta centrado en los procesos (ejemplo proceso de compra de viajes online) se tiende a elegir arquitecturas de servicios web, cuando el problema esta centrado en los recurso (por ejemplo el manejo de maquinas virtuales) se adecuan mas las arquitecturas REST. Un argumento de REST, es que la web por propia naturaleza no es RPC, por tanto los servicios tampoco deberian ser implementados con este procedimiento, aunque se dice que es un error arrastrado de la primera implementacion SOA que se llevo a cabo del mundo CORBA.

 

 

La principal ventaja es la simplicidad, ya que no es necesaria una defincion del contrato de servicio (lo que le hace plantearse si realmente es una arquitectura SOA) aunque existe el lenguaje WADL que permite definirlo y se usa directamente sobre los 4 metodos de HTTP.

 

 

En un mensaje HTTP tipicamente una serie de headers y si envia datos un body , en un request son todo cabeceras:

GET /hello.htm HTTP/1.1
								User-Agent: Mozilla/4.0 (compatible; MSIE5.01; Windows NT)
								Host: www.tutorialspoint.com
								Accept-Language: en-us
								Accept-Encoding: gzip, deflate
								Connection: Keep-Alive

								POST /enlighten/calais.asmx/Enlighten HTTP/1.1
								Host: api.opencalais.com
								Content-Type: application/x-www-form-urlencoded
								Content-Length: length
								licenseID=string&content=string&paramsXML=string

En un response :

 

HTTP/1.1 200 OK
								Content-Type: text/xml; charset=utf-8
								Content-Length: length
								<?xml version="1.0" encoding="utf-8"?>
								<string xmlns="http://clearforest.com/">string</string>

En cuanto al mundo JAVA, el estandar es denominado e implementado a traves de JAX-RS aunque existen otras implementaciones con funcionalidades adicionales como CXF, Axis o Jersey.
