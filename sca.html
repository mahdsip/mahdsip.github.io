SCA

Service Component Architecture

Es considerada como una implementación de SOA mas amplia que web services. La organización de desarrollo del estándar elegida fue OASIS. Proporciona un modelo abierto independiente de la tecnología para implementar IT services definidos como funciones de negocio y hace mas accesibles al desarrollador las funcionalidades middleware. Se integran componentes escritos en cualquier lenguaje de programación.

 

Los objetivos de SCA son:

Definir de manera única un modelo para la provisión y consumo de servicios de cualquier lenguaje

Definir de manera única el modo de ensamblar los componentes

 

El SCA Assembly Model consiste en una serie de artefactos definidos en un XML. El artefacto base es el Composite donde se describen los detalles del servicio así como sus llamadas a otros servicios que puedan ser necesarios.

 

SCA ofrece interoperabilidad en lugar de mediación (como JBI) o invocación (como JCA) de manera que es capaz de enlazar todo el flujo de trabajo sin un intermediario, todo el flujo de trabajo reside en cada uno de los extremos de manera que cada extremo sabe a quien tiene que llamar en siguiente lugar. Esto no es siempre así y muchas veces se hace uso de un ESB.

 

Esta tecnología va muy de la mano de SDO, que es una especificación diseñada para simplificar y unificar el acceso a la información ante los diferentes formatos y orígenes de datos existentes, con el objetivo de trabajar con una API única capaz de gestionar un acceso a bbdd, a un xml, a un servicio web, etc.

 

Las implementaciones mas conocidas son Apache Tuscany (Fabric3 es una implementacion lista para produccion) y Service Conduit, implementacion para sistemas windows.
