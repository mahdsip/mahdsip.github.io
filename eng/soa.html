SOA

Service Oriented Architecture

SOA development involves the following practices in architecture ( SOA Compliant by Thomas Earl ) :

  • Reusable components
  • Formal contract
  • Low coupling
  • Service composition
  • Autonomus services
  • Stateless snervices
  • Has public access

 

It is therefore a way to reach more than a product architecture. First you must create the architecture for later choose the platform where you want to run , as with web application servers or web services containers . With an SOA service granularity right business to the needs expressed by the business unit of the company sought ( ALIGNMENT , the key to success is good alignment between business and services ) .

he difference with object orientation O.Ö. It is that it is run by looking granularity processes in services. In the programming O.Ö. Objects are composed by their state (attributes), behavior (methods) and their unique identifier. Its basic principles are

Modularity: Divide the functionality between objects between well-defined modules

Encapsulation: Objects expose an interface to interact with each other and conceal details of their implementation. In reference to this there are two important concepts:

Cohesion: the object is all you need for yourself and all its contents is directly relaccionado the abstraction of the object. It is desirable therefore very cohesive

Coupling: The need to have to know the details of an object other object. Low coupling is desirable.

Disguise: Protects the attributes of a class that can not be changed from the outside, or set setter and public getters to manipulate

Heritage: an object can inherit state and behavior of another.

Polymorphism: Possibility of implementing the same functionality differently depending on the object name that is running.

 

They are actually very similar styles, both seek much cohesion and coupling bit but in this case referred between services, not the object's methods.

 

    Granularity: Fine Objects → Services → Thick
    Contract Objects Public / Private, Public Services
    Objects Reusability: medium, high Services
    Objects Unit: Compilation, services: runtime.

 

There are different implementations of SOA in three categories:

 

- File: Both request and response files are exchanged. They are Web services or SCA examples. When communicating systems are not exchanging data with RPC, they are being exchanged documents (eg XML). Services are not interested in knowing the interface itself (as in the case of the PRC) and if they are in the contract of the things they have to do that between the documents are exchanged.

- Message: such as RPC. Examples of implementation are CORBA or COM in which the message content is sent in binary. COM and CORBA would become later in DCOM and RMI to support the call to objects outside the network. They never reach the levels of interoperability of the other two types.

- Resource: This is making requests to resources that files sometimes and sometimes not exchanged. The best known implementation is REST.Los resources (data elements) that can be accessed and identified using a global identifier (Uniform Resource Identifier) ​​to be used as an identifier within the message each time you want to refer to it. For REST came to be considered an SOA need to have a defined contract, otherwise what would SOA would qualify to become a resource-based architecture.

 

For example, an SOA implementation through web servicices would combine SOAP as communication protocol and message format XML, WSDL as a service contract.

 

SOA can be fully descencentralizada, as in implementations through SCA or have a middleware to help themselves in a work arquitecura services by shifting responsibility for this work from the ends to a central body. One of these responsibilities may be for example the composition of services, as mentioned above, it is one of the foundations of these architectures. Other RESPONSIBILITIES can be routed, message transformation, service discovery, etc.
