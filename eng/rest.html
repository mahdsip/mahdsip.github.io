REST

Representational State Transfer

It is a type of resource-based architecture, even if it is accompanied by the definition of a service contract can be considered a type of SOA resource type. It is used to describe any simple web interface that uses XML and HTTP typically. It is based on the existence of accessible resources THROUGH URIs. To manipulate resources via HTTP communicate and exchange reprsentaciones of these resources. RPC you have operations you can perform, you get the REST resource (XML or other leguaje) and would operate on then upload it to an HTTP put. REST involves changing the state of a resource but without defining the operation. The requests can be made to a resource they are limited by HTTP: POST for storing data, to collect GET PUT resource to modify and DELETE to delete it. When the problem we try to solve this issue focused on processes (eg purchasing process online travel) tend to choose Web services architectures, when the problem is centered in the resource (eg managing virtual machines) are suitable more REST architectures. An argument of REST is that the web by nature is not RPC therefore services should also not be implemented with this procedure, though it is said to be a creep error of the first SOA implementation that took place in the world CORBA.
